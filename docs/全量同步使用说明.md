# 全量笔记同步使用说明

## 概述

全量同步功能用于将笔记仓库的所有笔记完全重新转换成博客文章，确保博客与笔记仓库完全一致。

## 使用场景

### ✅ 适合使用全量同步的情况：

1. **大量笔记更新**：笔记仓库中有大量文件被修改
2. **格式统一**：需要确保所有笔记都有完整的 front matter
3. **修复问题**：博客文章出现格式问题，需要重新生成
4. **初次设置**：首次从笔记仓库导入所有笔记

### ❌ 不适合使用全量同步的情况：

1. **博客中有手动优化的内容**：全量同步会覆盖所有文件
2. **只有少量更新**：使用增量同步更快速
3. **博客有独立内容**：博客中存在笔记仓库中没有的文章

## 使用方法

### 1. 增量同步（默认）

```bash
./deploy.sh
```

**行为**：
- 只添加新笔记
- 不更新已存在的博客文章
- 后台运行，不阻塞部署

**适合**：日常使用，只添加新笔记

### 2. 全量同步

```bash
./deploy.sh --full-sync
```

**行为**：
- 确保笔记仓库与远程同步
- 备份现有博客文章到 `/tmp/blog-post-backup-*`
- 删除所有存量博客文章（保留 `_index.md`）
- 从笔记仓库重新复制所有笔记
- 为所有文件添加/更新完整的 front matter

**适合**：大量更新或需要完全重建时

## 全量同步流程

```
【步骤 1/5】确保笔记仓库与远程一致
  ↓
  - 检测未提交的修改并提交
  - 拉取远程最新代码
  - 推送本地提交到远程

【步骤 2/5】备份现有博客文章
  ↓
  - 备份到 /tmp/blog-post-backup-YYYYMMDD-HHMMSS

【步骤 3/5】全量删除存量博客文件
  ↓
  - 删除 content/post/ 中所有 .md 文件
  - 保留 _index.md

【步骤 4/5】全量复制笔记到博客
  ↓
  - 从笔记仓库复制所有 .md 文件
  - 排除：.git/, images/, assets/, .obsidian/, 工作/

【步骤 5/5】添加/更新 front matter
  ↓
  - 使用 GitHub API 获取精确时间戳（如果有 GITHUB_TOKEN）
  - 为所有文件添加完整的 front matter
  - 补充缺失的必填字段（title, categories, date, draft）
```

## Front Matter 处理逻辑

### 对于已有 front matter 的文件：

```yaml
---
title: 保持原标题或从内容提取
categories: 保持原分类或从目录推断
date: 保持原日期或使用文件修改时间
draft: 保持或默认为 false
---
```

### 对于没有 front matter 的文件：

```yaml
---
title: 从文件内容第一个 # 标题提取
categories: 从目录结构推断（如 AIGC学习笔记/ → ['AIGC学习笔记']）
date: 使用文件修改时间（东八区）
draft: false
---
```

## 备份与恢复

### 自动备份位置

```bash
/tmp/blog-post-backup-YYYYMMDD-HHMMSS/
```

### 手动恢复备份

```bash
# 查看所有备份
ls -la /tmp/blog-post-backup-*

# 恢复特定备份
cp -r /tmp/blog-post-backup-20260306-180000/post/* content/post/
```

## 常见问题

### Q1: 全量同步会删除笔记仓库的文件吗？

**A**: 不会。全量同步只操作 `content/post/` 目录，不会修改笔记仓库。

### Q2: 如果同步后发现有问题怎么办？

**A**: 可以从自动备份中恢复：
```bash
# 查找最近的备份
ls -lt /tmp/blog-post-backup-*

# 恢复备份
LATEST_BACKUP=$(ls -td /tmp/blog-post-backup-* | head -1)
cp -r "$LATEST_BACKUP/post"/* content/post/
```

### Q3: 全量同步需要多长时间？

**A**: 取决于笔记数量：
- 100 篇笔记：约 30-60 秒
- 包含 GitHub API 查询：额外 1-2 分钟

### Q4: 能否只同步特定目录的笔记？

**A**: 当前版本不支持。如果需要，可以修改 `scripts/full-sync-notes.sh` 中的 rsync 参数。

### Q5: 增量同步和全量同步可以混用吗？

**A**: 可以。建议：
- 日常使用增量同步：`./deploy.sh`
- 定期（如每周）使用全量同步：`./deploy.sh --full-sync`

## 最佳实践

### 推荐工作流

```bash
# 场景 1: 创建新笔记
cd /root/.openclaw/workspace/note-gen-sync
vim 新笔记.md
cd /var/www/bluespace3.github.io
./deploy.sh  # 增量同步即可

# 场景 2: 修改大量已有笔记
cd /root/.openclaw/workspace/note-gen-sync
# ...修改多个笔记...
cd /var/www/bluespace3.github.io
./deploy.sh --full-sync  # 使用全量同步

# 场景 3: 只修改博客中的特定文章
cd /var/www/bluespace3.github.io
vim content/post/特定文章.md
./deploy.sh  # 不需要同步笔记
```

### 注意事项

1. **定期备份**：虽然脚本会自动备份，但建议定期手动备份整个博客
2. **测试后使用**：首次使用时，建议在测试环境验证
3. **查看日志**：同步后查看日志，确认没有错误
4. **保留备份**：不要删除 `/tmp` 中的备份，至少保留最近 7 天

## 技术细节

### Front Matter 必填字段

所有博客文章必须包含以下字段：

```yaml
title: 文章标题（必需）
categories: 分类列表（必需）
date: ISO 8601 格式日期（必需）
draft: true/false（必需）
```

### 时间戳优先级

1. **最高优先级**：GitHub API 创建时间（需要 GITHUB_TOKEN）
2. **中等优先级**：文件已有 front matter 中的 date 字段
3. **最低优先级**：文件修改时间（mtime）

### 目录到分类的映射

```
content/post/AIGC学习笔记/RAG.md
→ categories: ['AIGC学习笔记']

content/post/实践/wsl.md
→ categories: ['实践']

content/post/root-level.md
→ categories: ['技术']  # 默认分类
```
