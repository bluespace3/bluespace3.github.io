技术开发经验#

  • 默认路径陷阱：执行"查找所有仓库"任务时，应优先读取全局 Git 配置的 recentrepo 记录来快速锁定高概率目录
  • 作者名匹配局限性：仅通过单一作者名筛选提交可能遗漏，需检查各仓库的 .git/config 中 user.name 的历史配置
  • AI 误判根因：单纯基于日志文本分析会导致上下文缺失，必须结合结构化的"失败类型"字段进行多维判定
  • 测试数据污染：测试环境特有的文案（如"测试"、“重复”）会干扰 AI 模型，需在 prompt 中明确加入排除规则或白名单机制
  • 批量操作安全：数据库批量操作前应严格使用 dry-run 预览机制，确认影响范围后再执行写入
  • 规则短路的重要性：高频且明确的结构化错误必须在送入 LLM 前被规则拦截，否则会导致大量无效请求消耗 Token

隐私安全#

  • 隐私安全需双重兜底：仅依赖 LLM 遵循脱敏指令存在风险，必须在程序写入前增加硬编码的正则清洗步骤
  • 提取字段依赖一致性：标题必须像正文一样由 LLM 归纳生成，不能机械地截取原始字段
  • 保留技术细节的重要性：在提炼过程中需保留"文件路径、命令、错误现象、验证结果"等具体信息，特别是根因分析与修复方案

流程规范#

  • 子代理流程顺序问题：工作流规范需严格限制收尾话术的触发时机，确保在"全部子代理结束、merge + CSV 同步完成后的最后一轮回复"展示
  • 业务定义的模糊性：在接手需求时，名词（如「环境问题」）的定义可能存在歧义，必须结合具体场景进行精确界定
