# 2026-07-16 赛博日记

## 重要事件

- 修复了 Git 仓库提交者身份被错误覆盖的问题，通过 `filter-branch` 重写历史提交的作者信息
- GitLab 分支保护暂时开启 force push 权限，完成推送后建议恢复保护

## 关键决策

- 选择使用 `git filter-branch` 而非 `filter-repo` 重写 Git 历史（filter-repo 需要额外安装，filter-branch 内置）
- 强推前创建 `backup-before-rewrite` 分支作为回退保险

## 技术笔记

### Git 身份配置层级优先级
```
本地配置 > 全局配置 > 系统配置
```
- `.git/config` (本地) 会覆盖 `~/.gitconfig` (全局)
- 检查命令：`git config --list --show-origin`

### Git 历史重写
```bash
# 按邮箱过滤重写作者
git filter-branch --force --env-filter '
if [ "$GIT_COMMITTER_EMAIL" = "错误的邮箱@example.com" ]; then
    GIT_COMMITTER_NAME="新名字"
    GIT_COMMITTER_EMAIL="新邮箱@example.com"
fi
if [ "$GIT_AUTHOR_EMAIL" = "错误的邮箱@example.com" ]; then
    GIT_AUTHOR_NAME="新名字"
    GIT_AUTHOR_EMAIL="新邮箱@example.com"
fi
' --tag-name-filter cat -- --branches --tags
```

## 学到的教训

- Git 提交一旦生成，作者信息永久写进 commit 对象，无法通过改配置追溯修改
- GitLab `main` 分支默认受保护，禁止强推，需要在网页 Settings → Repository → Protected Branches 开启权限
- 重写 Git 历史会改变所有 commit hash，需要强推覆盖，多人协作场景下风险极高
