## 💡 深度洞察与经验教训

- **洞察：Swagger 解析器的健壮性是 API 测试工具的基础**
  - 循环引用和深度溢出是 OpenAPI 文档解析的常见陷阱
  - 需要在设计阶段就考虑防御性编程，包括引用栈追踪和深度上限控制
  - 对于复杂文档（88 个 POST 接口、253 个 components schema），这种防护机制至关重要

- **洞察：LLM 的不稳定性需要多层容错机制**
  - LLM 输出 JSON 格式不稳定，需要实现去 markdown、修复尾随逗号、截断补齐等容错逻辑
  - 单一依赖 LLM 有风险，自动降级到 Rule 模式是关键的兜底策略
  - 请求头序列化要求严格，整数如 `0` 会导致 `InvalidHeader` 错误

- **教训：FastAPI 204 返回的陷阱**
  - DELETE 路由返回 204 时，不能有返回类型注释 `-> None`
  - 不能使用 `response_class=Response` 配合 `return Response(status_code=204)`
  - 这是框架层面的限制，需要在设计 API 时特别注意

- **教训：数据持久化的系统性思考**
  - 任务列表存储在内存中，重启后端会清空
  - 需要持久化的数据（如认证配置）应写入磁盘文件
  - 未来应考虑任务元数据持久化到 `data/tasks.json`

- **教训：国内开发环境的特殊挑战**
  - Electron 二进制文件下载超时是常见的网络问题
  - 使用 npmmirror 镜像是有效的解决方案
  - 需要在基础设施层面考虑网络环境差异
